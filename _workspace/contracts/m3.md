# M3 계약 — RECO (조건 설정·비동기 AI 분석·추천 결과)

공통 규칙은 `api-contract-guide.md` 표준을 따른다 (Base URL `/api/v1`, ApiResponse 포맷, camelCase, `Authorization: Bearer {accessToken}`). 모든 엔드포인트 **인증 필요**, 소유자 본인만 접근(타인 리소스 → AUTH_003).

## 범위 및 결정사항

- FR-PREF-001~004, FR-PREF-006(생활방식 설문), FR-RECO-001~012, FR-RECO-014(추천 이유), FR-RECO-015(적합도 점수)
- 화면: RECO-001/002 조건설정(_10), RECO-003 분석진행(ai_2), RECO-004 요약(_4), RECO-005 상세(전용 폴더 없음 — _4 섹션구조 + IA 8섹션), RECO-006 전후비교(_5), RECO-007 가구상세(_17), RECO-008 전문가확인(_11)
- **결정 1 (비동기 잡)**: 분석은 `@Async` + 별도 Executor. POST /analyses는 즉시 **202 + analysisId** 반환, 클라이언트는 GET /analyses/{id} 폴링(5초). 동기 대기 금지.
- **결정 2 (Mock AI 단계 진행)**: `AiAnalysisService` 인터페이스 + Mock 구현. Mock은 단계별로 상태·진행률을 시간차로 갱신(QUEUED→ANALYZING_STRUCTURE→ANALYZING_LIGHT→ANALYZING_FLOW→GENERATING_RECO→GENERATING_VISUAL→COMPLETED). 각 단계 약 2~4초, 총 15초 내외로 시뮬레이션(테스트 가능하도록 짧게). M2의 `AiEstimationService`와 동일 패턴.
- **결정 3 (추천안 생성)**: 선택 스타일 수(1~3)만큼 Recommendation 생성. 각 추천안은 8개 섹션 + 새 필드(reason/fitScore/실제 제품형 items)를 Mock으로 채움.
- **결정 4 (적합도 점수는 실제 룰)**: `fitScore`는 Mock이 아니라 **실제 산술 계산** — M2에 저장된 공간 치수(widthM/depthM) vs 추천 가구 치수(widthMm 등)로 통로 폭·벽 길이 대비를 계산해 GOOD/CHECK/BLOCKED 판정. 이 로직은 단위 검증 대상(테스트 남길 것).
- **결정 5 (시각화)**: 실제 이미지 생성 없음. `visuals`의 afterUrl은 스타일별 정적 플레이스홀더, beforeUrl은 M2 업로드 원본 사진. regenerate는 스텁(202만).
- **결정 6 (RECO-007 가구 상세)**: 추천안 상세 응답의 `items[]` 항목을 그대로 사용(별도 엔드포인트 없음). FE가 itemId로 클라이언트에서 상세 표시.
- ID는 M1/M2와 일관되게 Long 자동증가.

## 엔드포인트

### POST /analyses

- **기능**: FR-RECO-001 공간 분석 비동기 실행
- **요청**:
  ```json
  { "spaceId": 1, "styles": ["NORDIC", "WOOD"], "budgetRange": "R300_500",
    "preferredColors": ["#F5F0E8", "#8B7355"], "requiredFurniture": ["SOFA", "BOOKSHELF"],
    "keepFurnitureIds": [20],
    "lifestyle": { "householdSize": 2, "hasChildren": false, "hasPets": true,
                   "worksFromHome": true, "cooksOften": false,
                   "storagePreference": "STORAGE", "housingType": "JEONSE", "residenceYears": 2 } }
  ```
  - `spaceId`: 본인 소유, 사진+치수 등록 완료 상태여야 함(미완료 → VALID_001)
  - `styles`: 1~3개(초과/미선택 → VALID_003)
  - `budgetRange`, `lifestyle` 필수. `preferredColors`/`requiredFurniture`/`keepFurnitureIds` 선택(nullable)
- **응답 202**:
  ```json
  { "success": true, "data": { "analysisId": 100, "status": "QUEUED", "estimatedSeconds": 15 } }
  ```
- **에러**: VALID_001(공간 데이터 불완전), VALID_003(스타일 수/enum), LIMIT_001(진행 중 분석 존재), AUTH_003, RES_001(공간 없음)

### GET /analyses/{analysisId}

- **기능**: FR-RECO-001 분석 상태 폴링 (RECO-003 진행 화면)
- **응답 200**:
  ```json
  { "success": true, "data": {
    "analysisId": 100, "status": "ANALYZING_LIGHT", "progress": 40,
    "currentStepLabel": "채광 분석 중...",
    "recommendationIds": [] } }
  ```
  - `status`가 COMPLETED면 `recommendationIds`에 스타일별 추천안 ID 목록(선택 스타일 순서)
  - FAILED면 `failureReason`(string) 포함, `progress`는 마지막 값
- **에러**: AUTH_003, RES_001

### GET /recommendations/{recommendationId}/visuals

- **기능**: FR-RECO-010 전후 비교 이미지 쌍 (RECO-006)
- **응답 200**:
  ```json
  { "success": true, "data": { "pairs": [
    { "beforeUrl": "/files/space1_photo.jpg", "afterUrl": "/files/placeholder_nordic_after.png", "viewLabel": "거실 전면" } ] } }
  ```
  - 원본 사진 삭제 시 `beforeUrl: null`(After 단독) + 응답에 `partial: true`
- **에러**: RES_001, AUTH_003, AI_004(시각화 생성 실패 — Mock 미발생)

### POST /recommendations/{recommendationId}/visuals/regenerate

- **기능**: FR-RECO-010 시각화 재생성(스텁)
- **응답 202**: `{ "success": true, "data": { "status": "QUEUED" } }`
- **에러**: RES_001, AUTH_003

## Enums

- StyleType: MODERN | MINIMAL | NATURAL | NORDIC | HOTEL | WOOD (M1과 동일)
- BudgetRange: UNDER_100 | R100_300 | R300_500 | R500_1000 | OVER_1000
- FurnitureType(category): M2 FurnitureType 재사용 (SOFA | BED | TABLE | CHAIR | WARDROBE | BOOKSHELF | TV_STAND | DRAWER | LIGHTING | ETC)
- JobStatus: QUEUED | ANALYZING_STRUCTURE | ANALYZING_LIGHT | ANALYZING_FLOW | GENERATING_RECO | GENERATING_VISUAL | COMPLETED | FAILED
- Verdict: GOOD | CHECK | BLOCKED
- StoragePreference: STORAGE | OPENNESS
- HousingType: JEONSE | WOLSE | OWNED

## 진행률 매핑 (Mock 참고, FE 진행 화면용)

| status | progress | currentStepLabel |
|--------|----------|------------------|
| QUEUED | 0 | 분석 대기 중... |
| ANALYZING_STRUCTURE | 20 | 구조·치수 분석 중... |
| ANALYZING_LIGHT | 40 | 채광 분석 중... |
| ANALYZING_FLOW | 60 | 동선 분석 중... |

## 적합도 점수 계산 규칙 (FR-RECO-015, 실제 구현)

- 입력: 공간 dimensions(widthM, depthM) + 추천 items의 치수(widthMm→m 변환)
- 검사 예:
  - **통로 폭**: depthM − (배치 가구 depth 합) ≥ 0.6m → GOOD, 0.4~0.6 → CHECK, <0.4 → BLOCKED
  - **가구 폭 vs 벽 길이**: item widthM ≤ widthM → 여유 계산, 여유 <0.2m → CHECK
- 총점: 검사 통과 비율 기반(GOOD=만점, CHECK 감점, BLOCKED 큰 감점), 0~100
- `measureBeforeBuy`: CHECK/BLOCKED 판정에 관여한 치수 항목명 수집
